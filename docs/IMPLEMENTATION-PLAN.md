# HDB Car Park Availability Dashboard — Implementation Plan

## Context

Build a new dashboard at `/Users/nilanka/Projects/hdb-carpark-availability-sg` for Singapore HDB carpark availability, mirroring the architecture and visual language of `traffic-dashboard-sg`. Two data sources from data.gov.sg are merged on the client:

- **Live availability**: `https://api.data.gov.sg/v1/transport/carpark-availability` (poll every 60s — per data.gov.sg recommendation).
- **Static carpark info**: dataset `d_23f946fa557947f93a8043bbef41dd09` (address, location coords in **SVY21**, carpark type, total decks, etc.). Stable inventory — fetch once.

User constraints worth noting:

- **~2,000 HDB carparks**. Showing all markers is feasible but visually noisy → use marker clustering at low zoom + color markers by availability ratio.
- **Address search is a primary discovery mechanism** (road name / building name / lane no). User picks from a result list → map flies to that carpark and opens its details panel.
- **Sharing strategy = copy-paste, diverge freely** (no shared package, no monorepo). Bootstrapped from traffic-dashboard-sg, then each app evolves independently.

The deployment story matches traffic-dashboard-sg: Vercel project with **Root Directory = `client`**, Vite preset, Serverless Functions under `client/api/*`, Express server at `server/` for local dev, `.env` for secrets (gitignored), localhost-only dev hosts (no LaunchAgent).

## New code

### 1. Static carpark info pipeline (`client/scripts/build-carparks.mjs`)

One-shot script (also via `npm run build:data`):

1. Read `docs/carpark-info-source.csv`.
2. Parse CSV with `papaparse`.
3. Convert `x_coord, y_coord` from SVY21 (EPSG:3414) → WGS84 (lat/lng) using `proj4`:
   ```js
   proj4.defs('EPSG:3414', '+proj=tmerc +lat_0=1.366666666666667 +lon_0=103.8333333333333 +k=1 +x_0=28001.642 +y_0=38744.572 +ellps=WGS84 +units=m +no_defs');
   ```
4. Emit `client/src/data/carparks.json`:
   ```ts
   { carparks: Array<{
       car_park_no: string,
       address: string,
       lat: number,
       lng: number,
       car_park_type: string,
       type_of_parking_system: string,
       short_term_parking: string,
       free_parking: string,
       night_parking: 'YES'|'NO',
       car_park_decks: number,
       gantry_height: number,
       car_park_basement: 'Y'|'N',
   }> }
   ```
5. **Commit both** `docs/carpark-info-source.csv` and `client/src/data/carparks.json`. User re-downloads + re-runs the script when the dataset changes (rare).

### 2. Serverless function (`client/api/carparks.js`)

```js
import { carparks, shapeCarparks, handleResource } from './_lib/upstream.js';
export default async function handler(_req, res) {
  await handleResource(carparks, shapeCarparks, res);
}
```

In `client/api/_lib/upstream.js`:

```js
export const carparks = createCachedFetcher({
  fetcher: () => fetchJson(`${apiBase()}/transport/carpark-availability`),
});

export function shapeCarparks(fetchedAt, payload, extra = {}) {
  const item = payload?.items?.[0];
  const rows = (item?.carpark_data ?? []).map((c) => {
    const info = c.carpark_info?.[0];
    return {
      car_park_no: c.carpark_number,
      total_lots: Number(info?.total_lots ?? 0),
      lots_available: Number(info?.lots_available ?? 0),
      lot_type: info?.lot_type ?? null,
      update_datetime: c.update_datetime ?? null,
    };
  });
  return {
    fetchedAt,
    apiTimestamp: item?.timestamp ?? null,
    count: rows.length,
    carparks: rows,
    ...extra,
  };
}
```

(Some carparks return multiple `carpark_info` entries — lot types C/Y/H. MVP takes the primary `C` entry; surface others in the panel if present.)

### 3. Client data join (`client/src/lib/mergeCarparks.js`)

```js
import staticInfo from '../data/carparks.json';
export function mergeCarparks(liveRows) {
  const byNumber = new Map(staticInfo.carparks.map((c) => [c.car_park_no, c]));
  return liveRows
    .map((live) => {
      const info = byNumber.get(live.car_park_no);
      if (!info) return null;
      return { ...info, ...live, availabilityRatio: info.total_lots > 0 ? live.lots_available / info.total_lots : null };
    })
    .filter(Boolean);
}
```

### 4. Search (`client/src/lib/searchCarparks.js` + `SearchBox.jsx`)

- `searchCarparks(query, carparks)` — case-insensitive substring match on `address` and `car_park_no`. No fuzzy lib for 2k items.
- `SearchBox` — controlled input + dropdown of up to 20 results. Each row: address, car_park_no, `lots_available / total_lots` badge. Click → `onSelect(car_park_no)`. Keyboard: ↑/↓, Enter, Esc.

### 5. Map (`client/src/components/MapView.jsx`)

- Use `leaflet.markercluster` `MarkerClusterGroup` instead of a plain `LayerGroup`.
- Marker icon: small badge with `lots_available` number; background colored by availability ratio:
  - `>= 0.5` → green (`--live`)
  - `0.1 – 0.5` → amber (`--stale`)
  - `< 0.1` → red (`--error`)
  - `null / 0 total_lots` → muted gray
- Selected marker: bigger + accent border.
- On `selectedId` change from parent: `map.setView([lat, lng], 17, { animate: true })`, open the marker's tooltip.
- Keep `HomeControl` (fly back to Singapore overview).

### 6. Right-side panel (`client/src/components/CarparkPanel.jsx`)

Same `aside.panel` shell as `CameraPanel`. Body:

- Carpark number, address
- Live lots available / total + percentage bar
- `update_datetime` (freshness)
- Car park type, parking system, decks, basement, gantry height
- Night parking / free parking flags
- If multiple `lot_type` entries on live row, list under "Other lot types"

### 7. App composition (`client/src/App.jsx`)
